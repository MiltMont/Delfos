# SCIP integration plan — complement tree-sitter content, by reference not by copy

## Context

Delfos builds its Cue→Tag→Content graph from a tree-sitter parse of each file
(`delfos/indexer/parser.py` → `extractor.py`). Separately, a full SCIP code-
intelligence index (`index.scip`) is checked in and a complete reader exists
(`delfos/scip/reader.py`), but it is **dead code** — never imported, never wired
into the indexer, store, or MCP server, and `index.scip` is generated by some
external tool with no step in the repo.

The goal: when an agent reaches a `ContentNode` during reconstruction, let it ask
SCIP-answerable questions about that node — *who references this, what does it
implement, what is its type definition* — **without duplicating** SCIP's cross-
reference data into the graph.

The chosen design is a **lazy bridge**:

- `index.scip` stays the single authoritative source of cross-references.
- Each `ContentNode` carries one new field, `scip_symbol`, a foreign key (the
  SCIP symbol string) — an identifier, not duplicated semantic content.
- New read-path MCP tools resolve that FK and query `ScipIndex` **on demand**.
  Nothing about references/implementations/types is materialized as graph edges.
- `index.scip` is **regenerated during `index`** so it tracks the code, and
  becomes a runtime artifact loaded alongside the store.

This fits the "reconstruction, not retrieval" thesis: cross-references become an
on-demand expansion the agent *chooses* to invoke, not pre-baked edges.

## Part 1 — Persist the `scip_symbol` FK on ContentNode

`ContentNode` round-trips through the C++ `NodeData` struct, so a new persisted
field must be threaded through every layer that `symbol_name` already passes
through. Mirror `symbol_name` exactly; it is a single optional string.

- **Schema** — add `scip_symbol: str | None = None` to `ContentNode`
  (`delfos/schema/nodes.py`, after `symbol_name` ~line 124, not at the class
  declaration on line 114).
- **C++ struct** — add `std::string scip_symbol;` to `NodeData` in the content
  block (`libdelfos/include/delfos/node.hpp:43`, next to `body`).
- **FlatBuffers** — **append** a `scip_symbol:string;` field to the NodeData
  table in `libdelfos/flatbuffers/delfos.fbs` (append-only keeps old snapshots
  readable — FlatBuffers gives new fields a default), then regenerate
  `libdelfos/flatbuffers/delfos_generated.h` with `flatc`.
- **Snapshot** — copy `scip_symbol` to/from the table in
  `libdelfos/include/delfos/snapshot.hpp`. In `to_fb_node` add
  `t.scip_symbol = n.scip_symbol;` after `body` (line 82); in `from_fb_node`
  add `n.scip_symbol = t.scip_symbol;` after `body` (line 109). Note: lines
  79–81 / 106–108 are the `symbol_name` block — `scip_symbol` goes after `body`,
  not alongside those fields.
- **Bindings** — add `.def_rw("scip_symbol", &NodeData::scip_symbol)` in
  `libdelfos/bindings/py_delfos.cpp` (near line 364).
- **Stub** — add `scip_symbol: str` to `NodeData` in `delfos/_delfos.pyi:62`.
- **Conversion** — in `delfos/store/native_store.py`, set
  `nd.scip_symbol = node.scip_symbol or ""` in the ContentNode branch of
  `_pydantic_to_native` (~line 152) and `scip_symbol=nd.scip_symbol or None` in
  the ContentNode branch of `_native_to_pydantic` (~line 209).

This field stays empty for `MODULE`-kind content (modules have no SCIP symbol).

## Part 2 — Generate SCIP and populate the FK at index time

SCIP indexes the whole repo at once, so generation + load is a single pre-pass
before the per-file loop; matching then happens per file inside the existing
transaction.

- **New module `delfos/scip/generate.py`** — a thin wrapper that shells out to
  the SCIP indexer: `scip-python index --project-name <root.name>
  --output index.scip --cwd <root>` (the `--output` path is relative to
  `--cwd`, so the file always lands at `root / "index.scip"`). The pipeline
  and the MCP server both load from `root / "index.scip"` — this must be the
  fixed canonical location. Run with the same `root` the indexer resolves so
  SCIP `relative_path` values match `path.relative_to(root).as_posix()`.
  Surface a clear error if the binary is missing from PATH; do not abort
  indexing — degrade to "no SCIP" (FK left empty).
- **Pipeline wiring** (`delfos/indexer/pipeline.py`): in `Indexer.index`, after
  resolving `root` (line 103), regenerate and load a `ScipIndex` once; pass it
  into `_index_file`. Keep it optional (`ScipIndex | None`) so indexing works
  when SCIP is unavailable.
- **Matching** — build a per-file map from SCIP **definitions**:
  `{occ.start_line + 1: occ.symbol for occ in scip.definitions(relative_path)}`.
  The key is `start_line + 1` (converting SCIP 0-based to delfos 1-based) so
  that lookups by `definition.lineno` are direct. The join is positional:
  tree-sitter sets `lineno = node.start_point[0] + 1` and SCIP places the
  definition occurrence at the same source line. This avoids parsing the SCIP
  symbol descriptor grammar.
- **Populate without breaking extractor purity** — `extractor.py` is
  deliberately I/O-free, so do **not** load SCIP there. Pass an optional
  `scip_symbols: Mapping[int, str]` (lineno → symbol) into `extract(...)` /
  `_Builder`, looked up in `_add_definition` (`extractor.py:122`) keyed on
  `definition.lineno`, and set on the `ContentNode` via a new `scip_symbol`
  parameter to `_add_content`. The pipeline computes that map from the loaded
  `ScipIndex` and hands it to `extract`.

Delete-and-reindex still holds: re-indexing a file rewrites its content nodes
(and their refreshed `scip_symbol`) from scratch.

## Part 3 — Read path: a ScipService + new MCP tools

- **New `delfos/scip/service.py`** — a `ScipService` holding both a
  `GraphStore` (to look up nodes by `content_id`) and a `ScipIndex` (to
  resolve the FK to cross-reference data), mirroring how `ReconstructionService`
  wraps the store. Constructor: `ScipService(store: GraphStore, index: ScipIndex)`.
  Methods take a `content_id`, fetch the node via the store, read its
  `scip_symbol`, and query the index:
  - `references(content_id)` → `ScipIndex.references(symbol)` (all usages,
    definition occurrence excluded — the caller already knows the definition
    site from the ContentNode).
  - `implementations(content_id)` → `symbol_info(symbol).relationships` filtered
    `is_implementation`.
  - `type_definition(content_id)` → relationships filtered `is_type_definition`.
  Return SCIP-native results (relative path + line range + symbol), which an
  agent can act on directly. **v1 scope:** do not attempt to resolve a
  referencing occurrence back to its enclosing `ContentNode` (that needs an
  enclosing-range index and a store-level `scip_symbol`→`content_id` reverse
  lookup that doesn't exist yet) — note as a future enhancement.
- **MCP tools** (`delfos/mcp/server.py`) — add thin `_`-prefixed functions and
  `@mcp.tool()` wrappers (`references`, `implementations`, `type_definition`)
  following the existing `_search`/`search` pattern, plus small view models in
  `delfos/mcp/views.py` for the returned occurrences. Wire a `ScipService` into
  `build_server` alongside `ReconstructionService` (load `index.scip` at server
  startup; if absent, the tools return an actionable "SCIP index not available"
  error rather than failing the whole server).
- Optionally extend the `reconstruct` prompt text to mention these tools as an
  expansion the agent may invoke once it reaches a content node.

## Risks / notes

- **Heaviest part is Part 1's plumbing**, but it is mechanical — a single string
  field mirroring `symbol_name` across known sites; low conceptual risk.
- **scip-python is an external (Node.js) dependency** — the binary is `scip-python`
  and must be on PATH (already installed at `~/.local/bin/scip-python` in this
  repo's dev environment). Generation must degrade gracefully when it's absent
  so indexing never hard-fails on it.
- **Snapshot compatibility**: only *append* to the `.fbs` table; never reorder or
  remove fields, or existing snapshots break.
- **Staleness window**: regenerating SCIP whole-repo while the indexer skips
  unchanged files is fine — the FK is only (re)written for files that are
  re-indexed, and a stale symbol simply resolves to fewer/no results, never a
  crash.
- **`reader.py` fixes already applied** (pre-implementation corrections):
  `_decode_range` now returns `(0,0,0,0)` instead of crashing on malformed
  occurrences that carry neither a typed nor a legacy range; `references()` now
  uses a pre-built inverted index (O(results) instead of O(all occurrences)) and
  excludes the definition occurrence from results.
