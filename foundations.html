MRAgent  /  Architecture Review

Foundations

The decisions that are blocking — things that cannot be deferred without forcing a rewrite later. Ordered by dependency: each item depends on the ones before it being settled first.

Dependency order

  • 1
    Cue–Tag–Content schema for code
    Everything else is a direct expression of this

    Shape determined by schema; no component touches DB directly

1
Code-specific Cue–Tag–Content schema
Why it comes first. Every other component depends on this. The DB schema, the indexer pipeline, the reconstruction algorithm, and the tool return types are all direct expressions of this schema. Building any of them without a settled schema means rebuilding them when it changes.

What must be decided

  • Cues: What constitutes a valid cue for code? Candidates: function names, class names, error message strings, concept phrases extracted by LLM ("rate limiting", "token refresh"). How are cues generated — purely from AST, from LLM extraction, or both? Are cues typed (symbol-cue vs concept-cue)?
  • Tags: What are the valid tag categories? Proposed set: module_path, arch_layer (api / service / data / infra), pattern_type (factory, middleware, singleton), lang_construct (decorator, context_manager, generic), language. Free-form strings or closed enum? Assigned by heuristic rules, AST analysis, or LLM labeling?
  • Content: What goes in a content node? Minimum: function/class body + signature + docstring. Extended: git commit messages that touched the symbol, associated test cases. Which are required at indexing time vs. lazily enriched?
  • Memory layers: What node types map to each layer?
  • Episodic Commits, PRs, code-change events. Granularity: per-file, per-symbol, or per-commit?
  • Semantic Exported API surface — interfaces, types, constants. How is "exported" defined across languages? How is semantic kept distinct from episodic for the same symbol?
  • Topic Modules and packages as architectural clusters. Cluster membership by directory structure, explicit annotation, or LLM clustering?
  • Edge types: Minimum: CUE_OF, TAGGED_WITH, PART_OF_TOPIC, REDIRECTS_TO (for renames). Are edges typed and directional? Can a cue belong to multiple content nodes?
Output A graph schema document with node types, required properties per type, edge types, and at least two worked examples (one function, one module) showing the full subgraph that would be created.
2
Database abstraction interface
Why it comes second. The tiered backend decision (SQLite + sqlite-vec for local, Neo4j for teams) means no component should ever call a database directly. The interface must be defined before the indexer or tools are implemented. Its shape is fully determined by the schema from step 1.

What must be decided

  • Interface boundaries: What primitive operations must the interface expose? Candidates: upsert_node, upsert_edge, vector_search(embedding, k), neighbors(node_id, edge_type, direction), get_node(id), delete_node(id), begin_transaction / commit / rollback. Are transactions first-class in the interface?
  • Vector search contract: Both SQLite-vec and Neo4j support ANN search with different guarantees. Does the interface promise exact k-NN or approximate? Return type: node_id + score, or full node?
  • Backend selection: Config file, environment variable, or URL scheme (neo4j:// vs sqlite://)? Is the backend swappable at runtime or fixed at startup?
  • SQLite-vec vs DuckDB: SQLite-vec is simpler to embed; DuckDB has a richer query language and handles larger graphs. This choice must be made now — the two have different transaction models.
  • Migration path: When a user moves from SQLite to Neo4j, what is the upgrade path? Does the interface include an export / import operation, or is re-indexing from scratch acceptable?
Output A Python abstract base class (GraphStore) with all method signatures, return types, and documented contracts. Both backend stubs written as skeleton classes satisfying the interface — no implementation yet.
3
reconstruct(query, budget) tool contract
Why it comes third. This is the primary interface between the MCP server and any consuming agent. The reconstruction algorithm, the tool's return type, and the trace format must all be specified before the server is built. The primitive tools (search, traverse_forward, traverse_reverse) are defined as a byproduct of this step.

What must be decided

  • Budget definition: What does budget control — number of traversal steps, total nodes visited, wall-clock time, or a combination? Sensible default? What happens when budget is exhausted — return partial results or error?
  • Traversal policy: The paper (Figure 9) shows sequential depth outperforms parallel breadth. Default order: depth-first, best-first by score, or iterative deepening? How does the algorithm decide when to switch from forward to reverse traversal?
  • Evidence set format: What does reconstruct return to the agent? Candidates: a list of content nodes with scores, a structured object with episodic/semantic/topic sections, a flat markdown summary. This directly determines how useful the tool is to an agent with no knowledge of the graph.
  • Reconstruction trace: Every call should log its traversal path. Structure: a list of (step, node_id, edge_type, score) tuples? Written to: log file, in-memory ring buffer, DB table? Included in the tool response (verbose mode) or only accessible via mragent inspect?
  • Primitive tool contracts: search(query) → List[CueNode], traverse_forward(cues, tags) → List[ContentNode], traverse_reverse(content_ids) → List[CueNode]. Return types must be consistent with what reconstruct uses internally.
Output A tool contract document with function signatures, argument types, return types, error conditions, and a worked example showing a full reconstruct call trace for a concrete query against a hypothetical codebase.
4
Stale-handling strategy
Why it comes fourth. The indexer is the write path. Writing it without a stale-handling strategy produces code that must be replaced entirely when stale content accumulates. The decisions here affect both the graph schema and the indexer logic. A codebase used for 6 months will have a graph full of ghosts if this is left unaddressed — and reconstruction quality degrades silently.

What must be decided

  • Node stamping: Every node must carry git_sha (the commit SHA at last index) and indexed_at (timestamp). These fields must be added to the schema from step 1 if not already present.
  • Symbol-level diffing: When a file changes, diff at the symbol level — not the file level. Old AST vs new AST; classify each symbol as: unchanged, modified, added, or deleted. Source of the "old AST": stored in the DB, re-parsed from git history, or diffed from git's object store?
  • Deleted symbols: A deleted function becomes a tombstone node with status: deleted, retaining its cues and edges. Metadata on tombstone: last known content, commit that deleted it, deleted_at timestamp.
  • Renamed symbols: A rename produces a REDIRECTS_TO edge from the old cue to the new. Traversal follows redirect edges transparently. Detection method: heuristic (same signature, different name), git blame, or explicit annotation?
  • Modified symbols: A significantly changed implementation needs re-extraction of cues and tags, not just content replacement. What threshold defines "significant" — any change, a diff exceeding N lines, or LLM-judged semantic distance above a threshold?
  • Merge semantics: The indexer must merge into the existing graph, not replace it. Merge algorithm: upsert by stable ID, upsert by (file_path, symbol_name), or content-addressed by hash?
Output A stale-handling spec with the merge algorithm, tombstone schema, redirect edge semantics, and a decision on how old ASTs are sourced for diffing.
5
Embedding model versioning
Why it comes fifth. Every node with a vector embedding is permanently tied to the model that produced it. This must be in the schema and the indexer from day one — adding it later requires re-reading every node to know which model was used. A half-migrated graph produces inconsistent similarity searches that are nearly impossible to debug.

What must be decided

  • Per-node metadata: Each node carrying an embedding must store embedding_model (e.g. text-embedding-3-small) and embedding_model_version (e.g. 2024-01). These fields must be added to the schema from step 1.
  • Model selection: Which model is used at indexing time — configured globally, per-project, or per-node-type? Is there a default that works without API keys (local model via sentence-transformers)?
  • Migration command: mragent migrate-embeddings --to <model> must be specified now even if implemented later. Re-embed all nodes in a single transaction, or is full re-indexing from scratch acceptable?
  • Cross-model search: If a partial migration is in progress, vector search almost certainly cannot work correctly. The DB backend must enforce that all nodes in a search index share the same model, and validate this invariant at index creation.
Output An addendum to the schema document specifying the embedding metadata fields, the model selection config format, and the migration command signature.
6
Indexer atomicity
Why it comes sixth. The indexer is the only writer. If it crashes mid-file, the graph is in an inconsistent state with no recovery path. Adding transactional guarantees after the fact requires rewriting the entire write path. There is currently nothing in the design preventing a half-indexed file from poisoning the graph.

What must be decided

  • Transaction granularity: What is the atomic unit — one file, one symbol, one commit? Per-file transactions are the practical choice: a file fits in a single transaction and the diff unit from git is already per-file.
  • Checkpoint manifest: A persistent manifest records which file SHAs have been fully committed to the graph. On restart, the indexer skips files whose SHA appears with status: committed. Format: SQLite table, JSON file, or dedicated Neo4j label?
  • Partial write recovery: If a transaction fails after some nodes are written but before commit — Neo4j: explicit transaction rollback. SQLite: automatic since transaction was never committed. Are there cases where auto-rollback is insufficient (e.g. vector index updates outside the transaction)?
  • Crash-safe manifest update: The manifest entry for a file must be written atomically after the DB transaction commits, not before. Mechanism: two-phase write, WAL-style approach, or treating the DB transaction itself as the manifest?
Output An indexer atomicity spec with the checkpoint manifest schema, the per-file transaction protocol, and the restart/recovery algorithm written in pseudocode.
These are specification tasks, not implementation tasks. Each produces a document or interface definition. The implementation plan cannot be written until all six exist — because every component (indexer, GraphStore backends, MCP server, CLI) is a direct translation of these specifications into code.
